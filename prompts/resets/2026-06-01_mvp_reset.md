## Project Vision

Mycelium is a LangGraph-based system for **AI-native, self-managing data sources**. The long-term goal is to create data infrastructure that is **100% managed by AI agents** rather than humans.

External AI agents interact with the system primarily through an **MCP server** using JSON.

## Current Phase: MVP (Phase 1)

We are in the early MVP stage with a deliberately **minimal core**.

### Core Architectural Decisions (Critical)

- **Everything is ultimately agent-owned** — including what we currently call the "core" CRM dataset.
- The **Supervisor** is a **coordinator and router**, not a data owner or direct accessor.
- Specialist agents will own both the responsibility *and* the storage strategy for their data domains.
- Even basic **identity resolution** (looking up a person by email, X handle, name, etc.) may eventually require querying specialist agents.
- We **do not pre-define** derivative datasets, attribute ownership, or storage structures. These emerge from the specialist agents.

### Core Data Model (Strictly Minimal)

- `Person` contains **only**: `id`, `name`, `employer`
- `CORE_PERSON_FIELDS = {"id", "name", "employer"}`
- `MINIMUM_VIABLE_FIELDS = ["name", "employer"]`
- All other attributes are treated as non-core and routed to specialist agents.

## Collaboration Model

- **Paul** directs the overall vision and priorities.
- **Grok (me)** acts as the planning, architecture, and review partner. I prepare high-quality instructions and structured prompts.
- **Cursor** acts as the senior developer that executes the implementation work in the IDE.

Work is directed to Cursor through structured, self-contained prompts placed in `prompts/cursor/next/`.

## Key Directory Structure

```
prompts/
├── cursor/                    # All Cursor-directed work
│   ├── next/                  # Ready tasks (timestamped .md prompts)
│   ├── in-progress/           # Currently claimed tasks
│   ├── done/                  # Completed work + full artifacts + reviews
│   ├── WORKFLOW.md            # Full protocol
│   └── README.md
├── grok-build/                # Prompts and context for Grok sessions
├── resets/                    # Context reset prompts (this file lives here)
└── system/
    └── CORE_PROMPT.md         # Canonical long-term project brief

docs/
├── architecture.md            # Current architecture and implementation direction
├── vision.md
└── ...

TODO.md                        # Long-term shared task list between Paul and Grok
```

**Critical Rule:** When Paul says "Work on the next task" (or similar), Cursor must:
1. Scan `prompts/cursor/next/`
2. Pick the oldest file (sorted alphabetically)
3. Move it to `prompts/cursor/in-progress/` (this is the claim)
4. Execute according to the prompt
5. Deliver complete artifacts to `prompts/cursor/done/`

## Must-Read Files at the Start of Every Session

1. `docs/architecture.md` — Current architecture and implementation direction.
2. `prompts/cursor/WORKFLOW.md` — How the Cursor handoff system works.
3. `TODO.md` — Current long-term priorities.
4. `prompts/system/CORE_PROMPT.md` — Foundational rules (no god-agents, explicit design, etc.).

## Working Principles

- **Scope discipline is mandatory.** Explicit scope boundaries in prompts must be respected. If out-of-scope work is needed, stop and escalate rather than proceeding.
- Prefer **simplification and deletion** over new abstractions.
- All changes should be **small and reviewable**.
- We treat Cursor as a senior developer: we give it clear, high-quality instructions and expect professional output + documentation.
- `docs/architecture.md` is the active source of truth for current implementation decisions.
